graphify — 把任意文件夹变成可查询知识图谱的 AI Coding Skill

项目概览

graphify 是一个 2026 年 4 月刚刚创建的 Claude Code Skill，仅用一周时间就斩获 19,600 Stars，定位非常清晰：把任意文件夹（代码、论文、截图、图片、PDF）转换为可交互查询的知识图谱。

创始人 Andrej Karpathy 有一个 /raw 文件夹，里面堆满了论文、推文截图和笔记。graphify 正是为了解决这个场景：持久化、结构化、可查询，且相比直接读原始文件，token 消耗降低 71.5 倍。

核心定位：让 AI Agent 能”看到”代码库的结构，发现跨文件的关联，并像人类一样通过浏览图来理解项目。

技术栈：

  • 语言分析：tree-sitter（多语言 AST 解析）+ call-graph 推理
  • 图计算：NetworkX（内存图）+ Leiden 算法（社区检测）
  • 多模态：Claude Vision（截图、图表、任意语言图片）
  • 可视化：vis.js（交互式 HTML 图）+ Obsidian 插件导出
  • 运行时：Python 3.10+，纯本地，无需 Neo4j 等外部服务

核心特性

1. 多模态输入：代码、文档、图片、PDF 全部支持

graphify 的提取策略根据文件类型动态切换：

类型 工具 提取内容
代码（15+ 语言） tree-sitter AST + call-graph 函数/类/变量节点 + 调用/导入边
文档（.md .txt） Claude 提取 概念节点 + 语义关联边
图片（.png .jpg .webp） Claude Vision 截图、图表、任意语言内容
PDF Claude 解析 引用挖掘 + 概念提取

最值得关注的是代码提取链路：先用 tree-sitter 生成 AST，再用 call-graph 第二遍推理，发现 AST 无法捕捉的间接调用关系，从而得到更完整的调用边（INFERRED 类型）。

2. 增量缓存：只重分析变化的文件

--update 模式利用 SHA256 缓存，只对变化文件跑 LLM 重提取，合并进已有图。这对大型代码库至关重要——Karpathy 的 52 个文件 corpus 每次全量跑完，graphify 能将后续查询 token 降低 71.5 倍。

3. 社区检测 + 上帝节点分析

graphify 不只是建图，还对图做进一步分析：

  • Leiden 算法（graspologic 库）做社区检测，发现紧密关联的模块集群
  • 上帝节点（God Nodes）：度最高的核心概念，所有其他节点的连接中枢
  • 意外连接（Surprising Connections）：代码-论文边的权重高于代码-代码边，结果带”为什么有关联”的自然语言解释
  • 建议问题：根据图结构生成 4-5 个图特别适合回答的问题

4. 多格式导出

  • graph.html：vis.js 交互式图，支持搜索和社区过滤
  • graph.svg：矢量图导出
  • graph.graphml：Gephi/yEd 格式
  • obsidian/：直接当 Obsidian vault 打开，用双链笔记方式浏览
  • --wiki：生成 index.md + 每个社区的 md 文章，Agent 可以通过读文件导航知识库
  • --neo4j：生成 Cypher 语句导入 Neo4j

关键工程设计

1. 信任标签：诚实告知”发现”还是”猜测”

每个边都带 confidence 标签，读取图的人始终知道信息来源：

标签 含义 示例
EXTRACTED 源码中明确存在 import 语句、直接函数调用
INFERRED 合理推断 call-graph 推理的间接调用
AMBIGUOUS 不确定，标记人工复核 高阶抽象关联

这解决了知识图谱中一个常见问题：图的质量不可知，用户盲目信任错误连接。

3. watch 模式：代码变化即时重建

  • 代码文件保存 → 触发即时 AST 重建（无需 LLM）
  • 文档/图片变化 → 通知运行 --update 做 LLM 重提取
  • 适合多 Agent 并行写代码场景，图在波次之间自动保持最新

4. MCP Server 模式

Agent 可以通过 MCP 协议直接查询图，把知识图谱当作记忆层使用。

benchmark 数据

Corpus 文件数 Token 压缩比
Karpathy repos + 5 papers + 4 images 52 71.5x
graphify 源码 + Transformer 论文 4 5.4x
httpx（Python 库） 6 ~1x（文件太少，上下文窗口够用）

Token 压缩比随语料规模非线性增长——6 个文件 fit 进上下文窗口，graph 的价值在于结构清晰度而非压缩。52 个混合文件时，71x 的压缩效果才真正体现。

架构亮点与借鉴

✅ 值得学习的点

  1. 纯函数 Pipeline：每个阶段通过 dict/Graph 通信，无隐式依赖，可独立测试
  2. Schema 强校验：extract → validate → build 链路保证了数据质量
  3. 增量语义缓存：SHA256 基础缓存 + LLM 重提取分层策略
  4. 信任标签设计：让下游消费者（人/Agent）知道置信度
  5. 多格式导出：同一个图，一次生成，多种使用方式（交互图/Obsidian/Wiki/矢量图）

⚠️ 潜在限制

  1. 依赖 Claude API：非代码内容（文档、图片）提取依赖 Claude，成本不可忽视
  2. 中文图片 OCR：Claude Vision 对中文截图的提取效果未经测试
  3. 大规模代码库：万级文件仍用单进程 NetworkX，可能有内存压力
  4. 社区检测质量：Leiden 算法对噪声边的敏感性未披露

总结

graphify 解决的是 AI Agent 时代的知识积累问题：当代码库、论文、笔记、截图混在一起时，怎么让 Agent 高效理解结构？

它的核心思路很聪明：不追求完美的语义理解，而是用”图”做结构化压缩 + 可视化，让 Agent 和人都能快速定位关键概念和意外关联。

项目刚创建一周就 19.6K Stars，说明这个需求确实广泛存在——不仅是 Claude Code 用户，OpenClaw、Cursor、Codex、Trae 用户都是潜在受众。
