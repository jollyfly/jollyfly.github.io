项目概览

graphify（safishamsi/graphify）是一个 AI 编程辅助 Skill，支持 Claude Code、Codex、Cursor、OpenClaw 等主流 Coding Agent。它将任意代码文件夹（code、docs、papers、images）转换为可查询的知识图谱，让你从结构层面理解一个代码库，而不是逐文件阅读。

⭐ 16,746 GitHub Stars | 2026-04-03 创建 | 7天暴涨 16K stars | Python | MIT 许可证

核心思想： Andrej Karpathy 有一个 /raw 文件夹收集论文、推文截图和笔记——graphify 就是这个问题的系统性答案：每次查询比读取原始文件少 71.5x tokens，且结果持久化到会话间。

核心特性

1. 两阶段提取：确定性 AST + LLM 语义理解

graphify 的 pipeline 分为两个完全不同的提取阶段：

第一阶段：确定性 AST 解析（零 LLM 调用）

使用 tree-sitter 解析代码的抽象语法树，提取：

  • 类、函数、方法定义（节点）
  • import/调用关系（边）
  • 继承关系（inherits 边）
  • 文档字符串和 rationale 注释（Python 专用）

这一阶段完全确定：相同代码每次运行结果一致，无 hallucination 风险。

第二阶段：LLM 子 Agent 语义提取（并行）

Claude 子 Agent 并行处理文档、论文、图片：

  • 提取概念和关系（semantically_similar_to 边，标记 INFERRED）
  • 提取设计意图（rationale_for 边）
  • 理解架构决策原因

两阶段结果合并进 NetworkX 图，统一做社区检测和导出。

2. 全模态输入支持

graphify 不只处理代码——它使用 Claude Vision 处理：

  • PDF 文档截图
  • 白板照片和架构图
  • 其他语言的截图（通过视觉提取概念）

3. 20 种语言的原生 AST 支持

通过 tree-sitter 实现对 20 种语言的原生解析：

语言 关键特性
Python 继承关系、Rationale 注释/docstring 提取
JavaScript/TypeScript Arrow 函数、类声明、方法定义
Go 方法接收者类型、方法体解析
Rust impl 块、trait、use 声明
Java/Kotlin/Scala scoped_identifier 解析、包作用域
C/C++ declarator 递归展开、头文件解析
Swift enum case、protocol 声明
Ruby/PHP/Lua 各种语言的特定语法
Objective-C @interface/@implementation/@protocol
Zig/PowerShell/Elixir/Julia 额外支持

4. 社区检测：纯图拓扑驱动，无需 Embeddings

这是 graphify 最独特的设计决策。 社区检测使用 Leiden 算法（优先）或 Louvain（fallback），基于图的边密度发现社区——不需要 embeddings，不需要向量数据库。

语义相似边（INFERRED 关系）已经存在于图中，直接参与社区检测——不需要额外的语义相似度计算步骤。

5. 关系可信度标签

每个关系边都带 confidence 标签：

标签 含义
EXTRACTED 源码中直接发现（import 语句、直接调用）
INFERRED 合理推断（调用图二遍、语义相似），带 confidence score
AMBIGUOUS 不确定关系，在 GRAPH_REPORT.md 中标记待审核

你永远知道什么是”找到的” vs “猜的”。

6. SHA256 缓存机制

首次运行后，graphify 对每个文件计算 SHA256 哈希。后续运行只处理哈希变化的文件，大幅加速增量更新。

适用场景与局限性

✅ 适用场景

场景 为什么适合 graphify
接手陌生代码库 快速理解模块划分和依赖关系，不用逐文件阅读
大型项目架构重构 通过图谱发现隐式耦合和高内聚低耦合的模块
代码审查 发现跨文件的隐式依赖关系
学习开源项目 从结构层面理解项目设计决策
团队知识传承 graph.json 持久化，新成员可直接查询而非重新探索
Graph RAG 配合 LLM 实现结构感知的代码问答

⚠️ 局限性

  1. tree-sitter 语言包依赖：未安装的语言包会报错而非静默跳过
  2. 动态语言解析不完整：Python 的 eval()、globals() 等动态特性无法静态解析
  3. 循环引用处理：调用图中的循环（a→b→c→a）会被 dedup 忽略，对分析有一定影响
  4. 多进程并行：缓存和提取框架支持并行，但 skill 层的 LLM 调用需要 Claude Code 的 subagent 支持
  5. LLM 成本：语义提取阶段的 tokens 消耗需要关注（但相比直接读源码已经节省 71.5x）

我的看法

技术价值：9/10

graphify 解决了 Code Agent 领域一个非常实际的问题：Agent 在处理陌生代码库时，只能通过 RAG 或逐文件读取来理解代码——这两种方式都无法捕捉代码的结构。

graphify 的核心洞察是：代码的结构（谁调用谁、谁继承谁、模块间关系）本身就是最有价值的知识，而这不需要 LLM 就能提取。tree-sitter 的确定性解析 + Leiden 社区检测，构成了一个完全可复现、可解释的图谱生成 pipeline。

最让我印象深刻的是它的增量设计：SHA256 缓存、跨文件 import 两遍解析、INFERRED confidence 标签——这些都是工程上非常成熟的设计，没有花哨的 AI 词汇，但每个细节都直击实际痛点。

对 Coding Agent 的意义：

如果你是 Claude Code / Cursor / OpenClaw 用户，/graphify . 应该是你接触任何新代码库的第一步——它给你的不是文件列表，而是一张架构地图。结合 GRAPH_REPORT.md 中的推荐问题，你可以快速建立对项目的心智模型。

对知识管理而言：

Andrej Karpathy 的 /raw 理念——把所有东西都扔进一个文件夹——是个人知识管理的反面。graphify 告诉你：信息之间的关系比信息本身更有价值。当你有了一个代码库的图谱，你可以问”哪些模块最核心？”、”这个设计决策的理由是什么？”——这些问题在文件列表里根本无从下手。
