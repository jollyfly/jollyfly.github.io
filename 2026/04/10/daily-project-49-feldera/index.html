项目描述： Feldera is a fast query engine for incremental computation. Feldera has the unique ability to evaluate arbitrary SQL programs incrementally, making it more powerful, expressive and performant than existing alternatives like batch engines, warehouses, stream processors or streaming databases.

传统的批处理引擎，每次数据变化都要从头重新计算——这就像每次房间乱了，都要从头整理整个房子。Feldera 的核心思想是：只处理变化的部分。当一张表新增了 100 条记录时，Feldera 只会重新计算受这 100 条记录影响的视图，而不是全量重算。

核心特性

  1. 真正的增量计算：基于 DBSP 理论（Dynamic Boolean Stream Processing），可以增量评估任意 SQL 查询，而不仅是简单聚合
  2. 百万级吞吐：单台笔记本可达数百万 events/second，无需调优
  3. 标准 SQL 支持：100% 标准 SQL 查询语法，支持递归查询（v0.31+）、窗口函数等高级特性
  4. 流批一体：用同一套引擎同时处理批数据和流数据，统一 batch + streaming 工作负载
  5. 丰富连接器：Kafka、HTTP、CDC、S3、Data Lakes、Warehouse 等多种数据源
  6. 递归 SQL：v0.31 支持递归 CTE，可处理图算法、层级遍历等场景

技术架构分析

整体架构

DBSP 理论基础

DBSP（Dynamic Boolean Stream Processing）是 Feldera 的数学基础，由斯坦福大学 M. B. Pawleski 和 R. R. Tood 开创。它的核心洞察是：

传统流处理的问题：大多数流处理系统只能处理受限的查询（如只支持 AVERAGE/SUM 等聚合），复杂 SQL（嵌套子查询、多表连接、窗口函数）的增量计算被认为是不可判定的。

DBSP 的突破：通过将所有数据表示为 Z-set（带权重集合，允许负权重表示删除），DBSP 可以为任意 SQL 查询生成增量评估算法。

当新交易插入时，Feldera 不需要重新扫描所有历史数据——它只计算新记录对 SUM 的影响，更新对应用户的 total。

适用场景与局限性

适用场景：

  • 实时 ETL：CDC 数据实时同步与转换
  • 特征工程：ML 模型的实时特征计算
  • 流式 OLAP：实时分析仪表盘
  • 渐变维度（SCD）：历史数据追踪
  • 物化视图维护：数据库的增量视图更新

局限性：

  • 生态还在早期（1.7k stars），生产案例有限
  • 需要额外学习 DBSP 思维模型
  • 分布式部署文档较少
  • 某些高级 SQL 特性（如某些窗口函数）尚不支持

我的看法

Feldera 解决的是一个很本质的问题：批处理太慢，流处理太弱。传统的数据仓库每次全量重算代价高昂，而流处理系统（Flink、Kafka Streams）支持的 SQL 表达能力有限。

DBSP 理论的价值在于它从数学上证明了”任意 SQL 都可以增量计算”，这比很多商业流数据库的”渐进式支持”要优雅得多。Rust 实现也保证了内存安全和性能。

对于需要处理频繁数据变更的场景（实时特征工程、CDC 数据处理、持续更新的物化视图），Feldera 是一个值得关注的技术选型。单台笔记本数百万 events/s 的性能在小规模场景下可以直接使用，生产规模也可以通过分布式部署扩展。
