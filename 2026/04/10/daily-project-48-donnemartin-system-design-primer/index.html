源码深度剖析

一、哈希表实现——链地址法的工业级写法

文件：solutions/object_oriented_design/hash_table/hash_map.py

关键设计决策：

  1. 链地址法（Separate Chaining）：每个桶是一个列表，解决哈希冲突。当链表过长时（>8），升级为红黑树——这是 Java 11 HashMap 的工业实现策略，本项目做了简化模拟。

  2. set() 的 UPSERT 语义：先遍历查找 key 是否存在——存在则更新，不存在才 append。这比每次都 append 再去重更高效，体现了读先于写的优化思路。

  3. KeyError 异常体系：用异常而非返回 None 来区分”未找到”和”值本身为 None”的边界情况，避免歧义。

复杂度分析：

操作 平均 最坏
get O(1 + n/k) O(n)
set O(1 + n/k) O(n)
remove O(1 + n/k) O(n)

其中 n=总元素数，k=桶数，n/k=平均链表长度。当 n/k 过大时需要扩容再哈希（Rehashing）。

二、LRU 缓存——双向链表 + 哈希表的经典组合

文件：solutions/object_oriented_design/lru_cache/lru_cache.py

这是系统设计中最重要的组合模式之一：哈希表 + 双向链表。

  • 哈希表：O(1) 按 key 查找节点
  • 双向链表：O(1) 在头部插入、在尾部删除，维护 LRU 顺序

为什么用双向链表而非单向？

单向链表只能 O(n) 找到前驱节点，无法在 O(1) 内完成 move_to_front（需要前驱指针才能更新链表指针）和 remove_from_tail（需要前驱节点才能断开）。双向链表的 prev/next 让这两个操作都变成真正的 O(1)。

工业级扩展思考：

  • 容量动态调整：当前 MAX_SIZE 固定；工业实现会考虑命中率低时自动扩容
  • TTL 支持：为每个 entry 加 timestamp 字段，淘汰过期数据
  • 写穿透（Write-Through）：set 时同步写后端存储，保证一致性
  • 分段 LRU：Redis 的 LFU 策略，将访问频率也纳入淘汰权重

三、停车场设计——面向对象设计的范本

文件：solutions/object_oriented_design/parking_lot/parking_lot.py

设计模式解析：

  1. 策略模式（Strategy Pattern）：每种车型（Motorcycle / Car / Bus）都实现了 can_fit_in_spot() 抽象方法。”能停什么车”的逻辑封装在车辆端，而非停车位端——体现了 Liskov 替换原则。

  2. 组合模式（Composite Pattern）：ParkingLot → Level → ParkingSpot 三层树形结构。顶层 API park_vehicle() 递归向下找合适车位，对外完全隐藏搜索细节。

  3. 尺寸枚举设计：VehicleSize.MOTORCYCLE=0, COMPACT=1, LARGE=2 用整数枚举，方便比较。Bus 需要连续 5 个大车位（spot_size=5），通过滑动窗口算法在一行内找连续空位。

四、呼叫中心——职责链模式（Chain of Responsibility）

职责链模式的核心： 每个员工只知道”处理或上抛”，不关心全局调度逻辑。

这个设计天然支持横向扩展——新增 MANAGER = 3 等级，只需在 dispatch_call 判断链中加入一行，无需改动 Employee 类。

五、MapReduce——分布式计算的核心范式

文件：solutions/system_design/mint/mint_mapreduce.py

MapReduce 的核心：分（Map）→ 排（Shuffle）→ 合（Reduce）。

为什么 MapReduce 在系统设计中重要？

它展示了用**无共享架构（Shared-Nothing）**处理海量数据：每个 Map 任务独立运行，只通过 key 通信。这使得系统可以水平扩展到数千台机器，任何一台机器宕机只影响它负责的数据分片，天然容错。
