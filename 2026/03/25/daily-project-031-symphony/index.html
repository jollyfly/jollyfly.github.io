daily project | Symphony 让团队以”管理工单”的方式驾驭 AI 编程 Agent

项目概览

Symphony（openai/symphony，⭐13,976）是 OpenAI 开源的项目级 AI Agent 编排框架。它的核心设计哲学一句话概括：把项目工作变成隔离的自主执行任务，让团队管理”工作”而不是监督”编码 Agent”。

传统开发流程中，工程师需要全程盯着 AI 编程助手：给它任务描述、审阅它写的代码、在 PR 里来回拉锯。Symphony 把这一切自动化：它像一个永不疲倦的高级工程师，持续监控 Linear 工单系统，将任务分配给编码 Agent（如 Codex CLI），等待 Agent 完成并验证 PR，最后将结果汇报给人类。整个过程无需人工干预，除非工单状态触发了特定的交接节点。

⚠️ 当前状态：Symphony 是 OpenAI 的低调工程预览版（low-key engineering preview），专为可信环境设计。官方建议在 adopt 了 harness engineering 理念的代码库中使用。

核心特性

1. 持续轮询 + 自主分发

Symphony 作为长时运行守护进程（daemon），以固定周期轮询 Linear 工单系统：

工单进入 active_states → Symphony 自动创建隔离工作空间 → 启动编码 Agent → 完成后提交 PR → 更新工单状态。

2. 隔离工作空间（Per-Issue Workspace）

每个工单分配独立的文件系统目录，编码 Agent 的所有操作仅限该目录内：

Agent Runner 在启动编码 Agent 前，会：

  1. 克隆仓库到 workspaces/{issue_identifier}/
  2. 基于工单描述 + WORKFLOW.md 模板渲染 system prompt
  3. 通过 JSON-RPC over stdio 启动 Agent（如 Codex CLI）
  4. 将 Agent 输出流式传回 Orchestrator

3. 仓库自管理的 WORKFLOW.md 契约

这是 Symphony 最具工程价值的理念：策略即代码，且策略归属团队。

团队可以在每个仓库中维护自己的 WORKFLOW.md，workflow 逻辑（Prompt、验证规则、交接策略）与代码库版本保持同步。

4. 可观测性（Observability）

Orchestrator 维护单权威内存状态，并通过结构化日志暴露：

运维人员可以通过日志监控多个并发 Agent 的运行状态、Token 消耗和错误率。

5. 自动恢复与重试

Symphony 内置了完整的错误恢复机制：

故障类型 处理策略
工单状态变更为非活跃（人工修改） 立即停止该工单的 Agent 运行
Agent 超时 进入重试队列，指数退避等待
瞬时网络错误 指数退避重试，最多重试 N 次
Workflow 文件缺失 抛出 missing_workflow_file 错误

重启后无需数据库，Orchestrator 内存状态通过文件系统 workspace 目录恢复。

关键设计决策

为什么是 Per-Issue Workspace 而不是共享分支？

因为工单之间的并行性不可预测。如果两个工单同时修改同一文件，per-issue workspace 确保各自的分支完全隔离。合并冲突的解决交给 Git 和 PR review 流程，而不是 Symphony 的协调逻辑。

为什么工单状态变更会导致 Agent 停止？

Symphony 的设计哲学是人类永远拥有最终决策权。如果工程师在 Agent 运行过程中手动将工单状态改为 Canceled，说明人工判断优先级已经改变，继续运行 Agent 是浪费资源且可能制造冲突。

为什么重试使用指数退避而不是固定间隔？

因为编码 Agent 的失败原因多样（LLM 服务端限流、网络抖动、仓库合并冲突），固定间隔容易多实例同时重试造成惊群效应（thundering herd），指数退避配合 jitter 可以有效分散重试压力。

适用场景

最适合

  1. 已采用 Harness Engineering 的大型代码库：团队已有成熟的 CI/CD、代码审查和分支策略，Symphony 在此基础上填补”AI Agent 编排”最后一环
  2. 多工单并行开发冲刺：一个 sprint 有 10+ 个工单，工程师不想逐一盯着每个 AI Agent，手工分配任务到各 Agent 的开销反而更高
  3. AI-First 重构项目：大量重复性代码迁移（如升级框架、批量重命名），Symphony 可以让多个编码 Agent 并行处理不同模块
  4. 与 Linear 深度集成的团队：Linear 已是团队工作流的中心，Symphony 无缝嵌入，不需要额外的任务管理系统

不适合

  • 小型项目（1-2 人团队，手把手 Code Review 比 Symphony 更快）
  • 没有 adopt harness engineering 的代码库（没有清晰的 CI、分支策略，Agent 输出难以验证）
  • 需要 Agent 实时交互的场景（Symphony 是批处理模式，不适合边对话边开发的场景）

总结

Symphony 的最大贡献不是技术实现，而是重新定义了人类工程师与 AI 编程 Agent 的协作模式：从”实时监督”到”异步工单管理”。工程师从”盯着 Agent 干活”转变为”验收 Agent 的工作成果”，大幅降低了人类注意力资源的消耗。

它的规范文档（SPEC.md）是目前最好的 AI Agent 编排系统设计参考之一，无论是否使用 Symphony，理解其中”Policy Layer 归属团队、Execution Layer 隔离、Coordination Layer 单一权威状态”的分层思想，都能帮助工程师设计更健壮的 AI 工作流系统。
